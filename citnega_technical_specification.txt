1. INTRODUCTION
================================================================================

Citnega is a local-first, extensible platform for running agentic workflows
through a conversational terminal interface. It is designed for general-purpose
tasks — research, writing, data processing, file operations — not coding
assistance.

The terminal UI is modeled after Gemini CLI and Claude Code: a single scrollable
conversation pane, streaming output rendered inline, tool calls and approvals
shown inline in the conversation flow, and slash commands for navigation.

The system supports two capabilities under one product umbrella:
  1. A headless agentic runtime for multi-step reasoning, tool use, session
     management, checkpointing, memory, retrieval, and knowledge organization.
  2. A conversational terminal UI over the runtime.

The critical architectural constraint: no layer above the adapter boundary
imports or references any specific agent framework (ADK, LangGraph, CrewAI).
The framework in use is a runtime configuration value. All frameworks are
equal. This is enforced by the package dependency matrix in Section 10.


================================================================================
2. PRODUCT GOALS AND NON-GOALS
================================================================================

2.1 Primary Goals
  - Conversational terminal-native application for general-purpose agentic work.
  - Single-pane UX mirroring Gemini CLI / Claude Code.
  - Runtime, TUI, CLI, model gateway, and KB fully decoupled from any framework.
  - Framework (ADK, LangGraph, CrewAI) is a config value, not an architectural bias.
  - Support both local and remote model providers with secure key storage.
  - Local knowledge management with full-text retrieval.
  - Core agents (orchestration) and specialist agents (domain tasks) share the
    same uniform structure as tools (CitnegaCallable hierarchy).
  - Cross-platform (Linux, macOS, Windows) with no platform-specific code in
    shared layers.
  - SOLID + DRY throughout.

2.2 Non-Goals for v1
  - No Kubernetes, containers, Redis, cloud object storage.
  - No distributed execution.
  - No PostgreSQL or enterprise infrastructure.
  - No vector databases (FTS5 only in v1).
  - No multi-panel TUI layout.
  - No DuckDB (deferred to analytics phase).
  - No entity extraction, topic detection, or KB semantic promotion (KB v2).
  - No plugins or dynamic loading in v1 (all callables registered at bootstrap).


================================================================================
3. SOLID AND DRY PRINCIPLES
================================================================================

3.1 Single Responsibility Principle
Every class has exactly one reason to change. Assignments:
  CitnegaCallable hierarchy   — callable contract only
  CallableRegistry            — register/lookup callables only
  PolicyEnforcer              — enforce execution policies only
  ContextAssembler            — assemble context only
  Each IContextHandler        — one enrichment step only
  IFrameworkAdapter           — one framework bridge only
  IFrameworkRunner            — one session's execution only
  ICallableFactory            — convert callables to framework form only
  ModelGateway                — route model requests only
  Each IModelProvider         — one provider API only
  ApplicationService          — service methods only; no business logic
  Each Repository             — persist one entity type only
  ArtifactStore               — filesystem artifacts only
  EventEmitter                — write events to queue only
  LogScrubber                 — strip sensitive fields only
  KeyStore                    — secret retrieval only
  Each SlashCommand           — one command only

3.2 Open/Closed Principle
Extension points (add new file; do not edit existing):
  New framework adapter   → implement IFrameworkAdapter + ICallableFactory, register
  New tool                → extend CitnegaTool, register in tool_registry.toml
  New specialist          → extend CitnegaSpecialist, register in agent_registry.toml
  New core agent          → extend CitnegaCoreAgent, register in agent_registry.toml
  New context handler     → implement IContextHandler, add to settings.toml
  New model provider      → implement IModelProvider, register in model_registry.toml
  New slash command       → implement ISlashCommand, register in SlashCommandRegistry
  New routing policy      → implement IRoutingPolicy, set in session config
  New event type          → subclass BaseEvent, no existing handlers break
  New repository          → implement IRepository[T], no other repos change

3.3 Liskov Substitution Principle
  - All IFrameworkAdapter impls must implement all lifecycle methods (pause,
    resume, cancel). Best-effort is acceptable; raising NotImplementedError is not.
  - IInvocable.invoke() must always return InvokeResult; never raise unhandled.
    Errors are captured inside InvokeResult.error.
  - IStreamable.stream_invoke() on non-streaming callables must yield one
    StreamChunk and then a terminal chunk. Never block indefinitely.
  - IModelProvider.stream_generate() must yield at least one chunk even for
    non-streaming backends.
  - IRepository[T].get() must return None for missing entities; never raise.
  - All three framework adapters must pass the same integration test suite.

3.4 Interface Segregation Principle
Callable hierarchy uses three narrow interfaces instead of one large one:
  IInvocable       — minimum contract (all callables)
  IStreamable      — adds stream_invoke (tools, specialists)
  IOrchestrable    — adds sub-callable management (core agents only)
Other segregated interfaces: IContextHandler, IRoutingPolicy, ISlashCommand,
IModelProvider, IRepository[T], IKeyStore, ITokenCounter. Each has only the
methods its callers need. No implementation should stub a method as unsupported.

3.5 Dependency Inversion Principle
Dependencies flow toward abstractions. Concrete types are only instantiated in
bootstrap.py (the composition root). Every component receives its dependencies
via constructor parameters. No module (except adapters/) imports a framework
package. No module (except bootstrap.py) imports concrete infrastructure.

Enforcement rule: any file other than adapters/*/ and bootstrap/ that imports
`google.adk`, `langgraph`, `crewai`, `ollama`, `openai`, or similar framework
packages is violating DIP and must be refactored.

3.6 DRY (Don't Repeat Yourself)
One canonical location per piece of logic:
  BaseCallable.invoke()   — pre/post logic (tracing, events, policy, errors)
  ContextAssembler        — context assembly pipeline
  PolicyEnforcer          — all policy checks
  PathResolver            — all path construction (wraps platformdirs)
  LogScrubber             — sensitive field scrubbing
  DatabaseFactory         — WAL initialization and connection setup
  BaseRegistry[T]         — shared register/resolve/list logic for all registries
  BaseEvent               — schema_version and timestamp assignment

5. FRAMEWORK-AGNOSTIC ADAPTER ARCHITECTURE
================================================================================

5.1 Principle of Equality
No framework receives preferential treatment above the adapter layer. Switching
frameworks is a config-only change. A session using ADK and a session using
LangGraph in the same process are identical at the service layer.

5.2 IFrameworkAdapter — the Port

```python
from abc import ABC, abstractmethod
from typing import Any
from citnega.protocol.models import Session, StateSnapshot
from citnega.protocol.callables import IInvocable
from citnega.protocol.events import CanonicalEvent

class AdapterConfig(BaseModel):
    framework_name: str                         # "adk" | "langgraph" | "crewai"
    default_model_id: str                       # model registry entry id
    framework_specific: dict[str, Any] = {}     # framework-specific options

class IFrameworkAdapter(ABC):
    @property
    @abstractmethod
    def framework_name(self) -> str: ...

    @abstractmethod
    async def initialize(self, config: AdapterConfig) -> None: ...

    @abstractmethod
    async def create_runner(
        self,
        session: Session,
        callables: list[IInvocable],
        model_gateway: "IModelGateway",
    ) -> "IFrameworkRunner": ...

    @abstractmethod
    async def shutdown(self) -> None: ...

    @property
    @abstractmethod
    def callable_factory(self) -> "ICallableFactory": ...
```

5.5 AdapterRegistry
BaseRegistry[IFrameworkAdapter] with register(name, adapter), resolve(name),
list_available(). bootstrap.py registers all available adapters once; the
runtime resolves the configured framework at session creation.

5.6 LSP Integration Test Suite
All three adapters must pass a shared test suite: `tests/adapters/shared_suite.py`.
Tests include:
  - Create session and run a simple turn with a mock tool.
  - Verify InvokeResult matches expected output for a deterministic test agent.
  - Verify streaming events arrive in correct order.
  - Verify pause/resume/cancel behavior (best-effort acceptable).
  - Verify checkpoint save and restore.
  - Verify event translation produces valid CanonicalEvent instances.
  - Verify policy enforcement errors propagate correctly.

Every new adapter added to the codebase must pass this suite before merging.

6. CALLABLE HIERARCHY
================================================================================

6.1 Interface Tree
  IInvocable                    ← all callables
    └─ IStreamable              ← tools, specialists
         └─ IOrchestrable       ← core agents only

6.2 IInvocable (minimum contract)

```python
from abc import ABC, abstractmethod
from pydantic import BaseModel
from typing import Type
from enum import Enum

class CallableType(str, Enum):
    TOOL = "tool"
    SPECIALIST = "specialist"
    CORE = "core"

class CallablePolicy(BaseModel):
    timeout_seconds: float = 30.0
    requires_approval: bool = False
    allowed_paths: list[str] = []           # empty = no file access allowed
    network_allowed: bool = False
    max_output_bytes: int = 256 * 1024      # 256 KB default
    max_depth_allowed: int = 2

class CallableMetadata(BaseModel):
    name: str
    description: str
    callable_type: CallableType
    input_schema_json: dict                 # JSON schema of input_schema
    output_schema_json: dict                # JSON schema of output_schema
    policy: CallablePolicy

class CallContext(BaseModel):
    session_id: str
    run_id: str
    turn_id: str
    depth: int = 0                          # invocation depth within a turn
    parent_callable: str | None = None
    session_config: "SessionConfig"
    model_gateway: "IModelGateway"          # injected for agents, not tools
    deadline: float | None = None           # monotonic clock deadline
    metadata: dict = {}

    class Config:
        arbitrary_types_allowed = True      # for IModelGateway reference

class InvokeResult(BaseModel):
    schema_version: int = 1
    callable_name: str
    callable_type: CallableType
    output: BaseModel | None = None
    error: "CitnegaError | None" = None
    duration_ms: int
    sub_invocations: list["InvokeResult"] = []
    token_usage: dict[str, int] = {}        # prompt_tokens, completion_tokens

    @classmethod
    def from_error(cls, name, type_, error, duration_ms): ...

    @classmethod
    def ok(cls, name, type_, output, duration_ms): ...

class IInvocable(ABC):
    name: str
    description: str
    callable_type: CallableType
    input_schema: Type[BaseModel]
    output_schema: Type[BaseModel]
    policy: CallablePolicy

    @abstractmethod
    async def invoke(self, input: BaseModel, context: CallContext) -> InvokeResult: ...

    @abstractmethod
    def get_metadata(self) -> CallableMetadata: ...
```

6.3 IStreamable (adds streaming)

```python
class StreamChunkKind(str, Enum):
    TOKEN = "token"
    TOOL_UPDATE = "tool_update"
    RESULT = "result"
    TERMINAL = "terminal"

class StreamChunk(BaseModel):
    schema_version: int = 1
    kind: StreamChunkKind
    content: str | None = None
    result: InvokeResult | None = None
    metadata: dict = {}

    @classmethod
    def token(cls, text: str) -> "StreamChunk": ...

    @classmethod
    def from_result(cls, result: InvokeResult) -> "StreamChunk": ...

    @classmethod
    def terminal(cls) -> "StreamChunk": ...

class IStreamable(IInvocable, ABC):
    @abstractmethod
    async def stream_invoke(
        self,
        input: BaseModel,
        context: CallContext,
    ) -> AsyncIterator[StreamChunk]: ...
```

6.5 BaseCallable (Template Method)

```python
class BaseCallable(IStreamable, ABC):
    def __init__(
        self,
        policy_enforcer: "IPolicyEnforcer",
        event_emitter: "IEventEmitter",
        tracer: "ITracer",
    ):
        self._policy_enforcer = policy_enforcer
        self._event_emitter = event_emitter
        self._tracer = tracer

    async def invoke(self, input, context):
        validated = self.input_schema.model_validate(input)
        await self._policy_enforcer.enforce(self, validated, context)
        self._event_emitter.emit(CallableStartEvent.from_invocation(self, context))
        start = time.monotonic()
        try:
            output = await self._execute(validated, context)
            result = InvokeResult.ok(self.name, self.callable_type, output,
                                     int((time.monotonic() - start) * 1000))
        except CitnegaError as e:
            result = InvokeResult.from_error(self.name, self.callable_type, e,
                                             int((time.monotonic() - start) * 1000))
        except Exception as e:
            wrapped = UnhandledCallableError(str(e), original=e)
            result = InvokeResult.from_error(self.name, self.callable_type, wrapped,
                                             int((time.monotonic() - start) * 1000))
        self._event_emitter.emit(CallableEndEvent.from_result(result, context))
        self._tracer.record(self, validated, result, context)
        return result

    @abstractmethod
    async def _execute(self, input: BaseModel, context: CallContext) -> BaseModel: ...

    async def stream_invoke(self, input, context):
        result = await self.invoke(input, context)
        yield StreamChunk.from_result(result)
        yield StreamChunk.terminal()
```

6.6 BaseCoreAgent

```python
class BaseCoreAgent(BaseCallable, IOrchestrable, ABC):
    callable_type = CallableType.CORE

    def __init__(self, *args, **kwargs):
        super().__init__(*args, **kwargs)
        self._sub_callables: list[IInvocable] = []
        self._routing_policy: IRoutingPolicy | None = None

    def register_sub_callable(self, callable): self._sub_callables.append(callable)
    def list_sub_callables(self): return list(self._sub_callables)
    def set_routing_policy(self, policy): self._routing_policy = policy
```

7. AGENT ARCHITECTURE
================================================================================

7.1 Tiers
  Core agents      Drive conversation, orchestrate sub-callables. Always present.
  Specialist agents Domain task execution. Invoked on demand.
  Tools            Atomic operations. Invoked by both.

7.2 Core Agents v1
  ConversationAgent  Primary orchestrator. System prompt template defines its role,
                     available specialists and tools, and synthesis style.
  PlannerAgent       Breaks complex multi-step tasks into ordered subtasks.
                     Invoked by ConversationAgent when task_complexity > threshold.

7.3 Specialist Agents v1
  ResearchAgent   web_search, kb_search, fetch_url, read_file
  SummaryAgent    (no tools — pure model operation)
  FileAgent       read_file, write_file, list_dir, search_files
                  (write/delete require approval; restricted to allowed_paths)
  DataAgent       read_file, parse_csv, parse_json, run_calculation
  WritingAgent    read_file, write_file

7.4 ConversationAgent System Prompt Template
Stored at packages/agents/core/prompts/conversation_agent.txt. Structure:
```
You are Citnega, a general-purpose agentic assistant running locally.
Your job is to understand the user's request, decide whether to invoke
specialist agents or tools, and synthesize a clear response.

Available specialist agents:
{specialist_descriptions}

Available tools:
{tool_descriptions}

Current session context:
{context_summary}

Rules:
- Use specialists for domain tasks (research, file operations, data analysis).
- Use tools only for atomic operations you can complete in one call.
- Ask for clarification if the request is ambiguous.
- Do not fabricate information. Cite sources from the knowledge base when used.
- Respect user approval prompts. Never bypass them.

User request: {user_input}
```

Placeholders are filled by the ContextAssembler from registry metadata at turn
start. The prompt template is read once per session (cached) and regenerated
only when callables are added/removed.

7.5 Routing
IRoutingPolicy selects sub-callables:
  LLMRoutingPolicy     LLM-driven via function calling (default)
  RuleBasedRoutingPolicy  Explicit regex/keyword patterns (rules take priority)
  HybridRoutingPolicy  Rules for high-confidence, LLM for the rest (recommended)

7.6 Depth Limits
  Depth 0  User input → Core agent
  Depth 1  Core agent → Specialist or Tool
  Depth 2  Specialist → Tool (maximum by default)
Configurable per session; enforced in PolicyEnforcer.DepthLimitCheck.

12. ASYNC ARCHITECTURE
================================================================================

12.1 Event Loop Ownership
  TUI process: Textual owns the asyncio loop.
  CLI process: asyncio.run() owns the loop.
  Never call asyncio.run() inside a Textual app.

12.2 Event Flow
```
IFrameworkRunner.run_turn()
    │
    └─ EventEmitter.emit() ──→ asyncio.Queue[CanonicalEvent] (maxsize=256)
                                    │
                                    └─ Textual Worker drains queue
                                           │
                                           └─ app.post_message()
                                                  │
                                                  └─ ConversationPane updates
```

12.3 Rules
  - All background tasks run as Textual Workers in the TUI.
  - Workers communicate to UI only via app.post_message() or call_from_thread().
  - Queue maxsize=256 applies backpressure to the framework runner.
  - RunCompleteEvent sentinel signals Workers to stop consuming.
  - No blocking I/O on the main loop; use aiosqlite, httpx async, asyncio.to_thread
    for keyring and filesystem operations when needed.

12.4 Concurrency Policy
  - One active run per session at a time. run_turn() is serialized per session
    via asyncio.Lock stored in SessionState.
  - Multiple sessions may run concurrently within one process.
  - KB ingestion and indexing run as separate Workers, independent of any session.

Error flow:
  - Lower layers raise typed errors.
  - BaseCallable.invoke() catches CitnegaError and wraps in InvokeResult.error.
  - Unhandled exceptions become UnhandledCallableError.
  - Service layer converts all errors into ErrorEvent for streaming.
  - TUI renders ErrorBlock. CLI prints and exits with code based on error_code.

Exit codes for CLI:
  0   Success
  1   Generic failure
  2   Config error
  3   Framework/adapter error
  4   Model gateway / no provider
  5   Storage / migration error
  64  Usage error (invalid CLI arguments)

17.3 Search Query (BM25 ranking)

```sql
SELECT
    i.*,
    snippet(kb_fts, 2, '[', ']', '...', 12) AS snippet,
    bm25(kb_fts) AS score
FROM kb_fts
JOIN kb_items i ON i.item_id = kb_fts.item_id
WHERE kb_fts MATCH ?
ORDER BY score
LIMIT ?;
```

19. BOOTSTRAP AND STARTUP SEQUENCE
================================================================================

bootstrap.py is the composition root — the ONLY file permitted to import concrete
types from all layers. Every other module receives its dependencies via
constructor injection.

Startup sequence (each step fails fast with a clear error):

  1.  Parse CLI args and environment variables.
  2.  Determine profile (dev/prod/default).
  3.  Load and validate all TOML config files via pydantic-settings.
  4.  Instantiate PathResolver (wraps platformdirs).
  5.  Create app directories if missing, apply permissions (0700 on Unix).
  6.  Instantiate DatabaseFactory, open connection, apply WAL PRAGMAs.
  7.  Run Alembic migrations (fail fast if schema is out of date).
  8.  Instantiate KeyringKeyStore (fallback EnvVarKeyStore).
  9.  Instantiate LogScrubber.
  10. Configure structlog; attach LogScrubber as processor.
  11. Instantiate all repositories (SessionRepo, RunRepo, KBRepo, ...).
  12. Instantiate ArtifactStore (PathResolver-based).
  13. Instantiate KnowledgeStore (wraps KBRepo + FTS5).
  14. Instantiate IModelProvider for each entry in model_registry.toml.
  15. Instantiate ModelGateway (inject all providers + HybridRoutingPolicy).
  16. Run ModelGateway.health_check_all(); warn on unhealthy providers.
  17. Instantiate EventEmitter and Tracer (inject repositories).
  18. Instantiate PolicyEnforcer with all checks.
  19. Instantiate ContextAssembler with handler chain from settings.toml.
  20. Instantiate all CitnegaTool from tool_registry.toml → CallableRegistry.
  21. Instantiate all CitnegaSpecialist from agent_registry.toml, inject tools.
  22. Instantiate all CitnegaCoreAgent, inject specialists + tools + routing policy.
  23. Instantiate all IFrameworkAdapter → AdapterRegistry.
  24. Call IFrameworkAdapter.initialize() on the configured framework only.
  25. Instantiate CoreRuntime, inject registries, gateway, context, policy, emitter.
  26. Instantiate ApplicationService, inject runtime, kb store, artifact store.
  27. Register ShutdownHandler for SIGTERM/SIGINT.
  28. Launch TUI or CLI depending on entry point (both receive ApplicationService).

Failure behavior:
  - Missing config file → exit code 2, clear error message.
  - Migration failure → exit code 5, clear error message, no data modified.
  - No healthy model provider + local_only=true → exit code 4.
  - No healthy model provider + local_only=false → warn and continue.
  - Framework adapter init failure → exit code 3, suggest config check.

20. RUN LIFECYCLE AND STATE MACHINE
================================================================================

20.1 RunState Transitions

```
                    ┌─────────┐
                    │ PENDING │
                    └────┬────┘
                         │ run_turn() called
                         ▼
                ┌────────────────────┐
                │ CONTEXT_ASSEMBLING │
                └─────────┬──────────┘
                          │ context ready
                          ▼
                  ┌───────────────┐
                  │  EXECUTING    │◄─────────────┐
                  └───┬───────────┘              │
                      │                          │
       ┌──────────────┼──────────────┐           │
       │              │              │           │
       ▼              ▼              ▼           │
┌──────────┐  ┌─────────────┐  ┌─────────┐      │
│COMPLETED │  │WAITING_APPR │  │ PAUSED  │──────┘
└──────────┘  └──────┬──────┘  └─────────┘   resume
                     │
          approved/denied/timeout
                     │
                     ▼
              ┌─────────────┐         ┌──────────┐
              │  EXECUTING  │────────▶│  FAILED  │
              └─────────────┘  error  └──────────┘
                     │
                cancel called
                     │
                     ▼
              ┌──────────────┐
              │  CANCELLED   │
              └──────────────┘
```

Transitions are managed by the RunLifecycleManager inside CoreRuntime. Each
transition emits a RunStateEvent. Invalid transitions raise InvalidRunStateError.

20.2 Concurrent Runs
  - One active run per session at a time. run_turn() on a session with an
    active run raises InvalidRunStateError.
  - Different sessions may run concurrently.

20.3 End-to-End Turn Flow

```
User types input into TUI
    │
    └─ ApplicationService.run_turn(session_id, input)
          │
          └─ CoreRuntime.run_turn()
                │
                ├─ 1. Acquire session lock
                ├─ 2. Create Run (state=PENDING)
                ├─ 3. Emit RunStateEvent(PENDING → CONTEXT_ASSEMBLING)
                ├─ 4. ContextAssembler.assemble()
                │       │
                │       └─ Run handler chain:
                │           RecentTurnsHandler → SessionSummaryHandler →
                │           KBRetrievalHandler → RuntimeStateHandler →
                │           TokenBudgetHandler
                ├─ 5. Emit ContextAssembledEvent
                ├─ 6. Emit RunStateEvent(CONTEXT_ASSEMBLING → EXECUTING)
                ├─ 7. AdapterRegistry.resolve(session.framework)
                ├─ 8. adapter.create_runner(session, callables, gateway)
                ├─ 9. runner.run_turn(input, context, queue)
                │       │
                │       └─ Framework executes core agent
                │           │
                │           ├─ Core agent may invoke specialists/tools
                │           │     │
                │           │     └─ Each invocation goes through:
                │           │         BaseCallable.invoke()
                │           │           │
                │           │           ├─ PolicyEnforcer.enforce()
                │           │           ├─ Emit CallableStartEvent
                │           │           ├─ _execute()
                │           │           ├─ Emit CallableEndEvent
                │           │           └─ Tracer.record()
                │           │
                │           └─ Stream TokenEvents into queue
                ├─ 10. Emit RunStateEvent(EXECUTING → COMPLETED)
                ├─ 11. Save run summary
                ├─ 12. Release session lock
                └─ 13. Emit RunCompleteEvent (queue sentinel)
```


================================================================================
21. CONTEXT ASSEMBLY PIPELINE
================================================================================

Implements Chain of Responsibility. Handlers injected via settings.toml
[context].handlers, run in the listed order, each enriches ContextObject.

21.1 Handlers (v1)

  RecentTurnsHandler
    Loads last N messages from the session (N = settings.context.recent_turns_count).
    Adds a ContextSource(source_type="recent_turns", content=formatted_messages).

  SessionSummaryHandler
    Loads the rolling summary from run_summaries table if present.
    Adds a ContextSource(source_type="summary").
    If no summary exists, skipped.

  KBRetrievalHandler
    Extracts keywords from user_input via a simple keyword extractor (no LLM).
    Runs FTS5 search, takes top N results (N = settings.context.kb_retrieval_limit).
    Adds ContextSource(source_type="kb") for each result.

  RuntimeStateHandler
    Loads current StateSnapshot for the session.
    Adds a ContextSource(source_type="state") with active callable and last outputs.

  TokenBudgetHandler (must be last)
    Uses the configured ITokenCounter (from current model's provider).
    Sums total_tokens across sources.
    If total > max_context_tokens: trims oldest sources (recent_turns first, then
    kb, then summary) until within budget.
    Sets context.truncated=True if trimming occurred.

21.2 Trimming Priority (when over budget)
  1. Drop oldest recent_turns one by one.
  2. Drop lowest-scoring kb results one by one.
  3. Truncate summary to fit remaining budget.
  4. If still over budget: raise InvalidContextError (should not happen in practice).


================================================================================
22. POLICY ENFORCEMENT
================================================================================

PolicyEnforcer runs a check chain before any IInvocable executes. Chain order:

  1. DepthLimitCheck   Rejects if context.depth > policy.max_depth_allowed.
  2. PathCheck         For file paths in input, validates against policy.allowed_paths.
                       Uses pathlib resolution; rejects symlinks escaping the allowlist.
  3. NetworkCheck      If tool attempts HTTP, validates policy.network_allowed.
                       (Enforced via tool's own network helper.)
  4. OutputSizeCheck   After execution, checks output size ≤ policy.max_output_bytes.
                       Raises OutputTooLargeError if exceeded.
  5. TimeoutCheck      Wraps _execute in asyncio.wait_for(timeout).
                       On timeout: cancels coroutine, raises CallableTimeoutError.
                       (Note: subprocess-based tools must handle SIGTERM themselves.)
  6. ApprovalCheck     If policy.requires_approval: creates Approval record, emits
                       ApprovalRequestEvent, transitions run to WAITING_APPROVAL,
                       awaits response (asyncio.Event), resumes on approval.
                       If timeout: raises ApprovalTimeoutError.
                       If denied: raises ApprovalDeniedError.

22.1 Cancellation Contract
  - asyncio.wait_for cancels the coroutine cleanly for pure-Python tools.
  - Tools that spawn subprocesses must register a cleanup handler via
    CallContext.register_cleanup(fn). BaseCallable calls all cleanups on timeout.
  - Tools that make HTTP requests must use httpx async client which respects
    cancellation.
  - Tools with file I/O must not hold locks across await points.

23. MODEL GATEWAY AND PROVIDERS
================================================================================

23.1 Routing Algorithm

```python
async def route(request: ModelRequest) -> IModelProvider:
    needs = request.needs or TaskNeeds(task_type="general")
    candidates = [p for p in self.providers if all([
        p.supports("streaming") if needs.streaming_required else True,
        p.supports("tool_calling") if needs.tool_calling_required else True,
        p.supports("reasoning") if needs.reasoning_required else True,
        p.model_info.local if needs.local_only else True,
        p.model_info.capabilities.max_context_tokens >= needs.min_context_tokens,
    ])]
    # Sort: preferred_for match first, then cost_rank, then priority
    candidates.sort(key=lambda p: (
        0 if needs.task_type in p.model_info.preferred_for else 1,
        p.model_info.cost_rank,
        p.model_info.priority,
    ))
    for candidate in candidates:
        status = await candidate.health_check()
        if status == "healthy":
            if self.rate_limiter.try_acquire(candidate.model_info.model_id):
                return candidate
    raise NoHealthyProviderError(...)
```

23.2 Tool Calling Normalization
Each provider receives a Citnega-native ModelRequest with `tools` in JSON schema
form. Each provider converts this to the format its API expects:
  - Ollama:       not yet standardized; use JSON mode + prompt engineering fallback
  - OpenAI-compat: native tools[] array with type="function"
  - vLLM:         OpenAI-compatible endpoint, same as above
The provider also normalizes responses back into ModelResponse.tool_calls form.

23.3 Rate Limiting
TokenBucketRateLimiter per provider, keyed on model_id. Buckets configured from
model_registry.toml requests_per_minute and tokens_per_minute. Rate limit hit:
  - If bucket has any tokens: wait until next token available.
  - If wait > request_timeout: raise RateLimitExceededError.
Emits RateLimitEvent when a wait occurs.

23.4 Token Counting
ITokenCounter implementations:
  - OpenAITokenCounter: wraps tiktoken with model-specific encoding
  - OllamaTokenCounter: uses tokenizers from transformers package if available,
                        else falls back to HeuristicTokenCounter (chars/4)
  - HeuristicTokenCounter: chars/4 approximation (last-resort fallback)

24. KNOWLEDGE BASE (v1)
================================================================================

24.1 Responsibilities (v1 minimal)
  - Store raw items (text, files, session excerpts, tool outputs, notes).
  - Maintain metadata: title, source_type, tags, timestamps, hash.
  - Index content in SQLite FTS5 with porter + unicode61 tokenizer.
  - Return ranked search results via BM25.
  - Support add, get, search, delete, list, export.

24.2 Ingestion Flow
```
IKnowledgeStore.add_item(item)
    │
    ├─ Compute content_hash (sha256)
    ├─ Check for duplicate by hash; skip if exists
    ├─ If content > 1 MB: write content to kb/raw/<item_id>.txt, store file_path
    ├─ Insert into kb_items table (trigger populates kb_fts)
    └─ Return KBItem
```

24.3 Search Flow
```
IKnowledgeStore.search(query, limit)
    │
    ├─ Sanitize query for FTS5 (escape special chars, quote phrases)
    ├─ Execute BM25-ranked SELECT with snippet() function
    ├─ Load full KBItem for each result
    ├─ If content was file-stored, read from disk
    └─ Return list[KBSearchResult]
```

24.4 v1 Explicit Non-Goals
  - No entity extraction.
  - No topic detection.
  - No embedding-based semantic search.
  - No automatic summarization of ingested items.
  - No candidate-index or promotion logic.
These are KB v2, behind the same IKnowledgeStore interface.

25. OBSERVABILITY AND LOGGING STRATEGY
================================================================================

25.1 Log Levels and Usage
  DEBUG    Developer diagnostics; disabled by default. Example: tool input/output dumps.
  INFO     Notable lifecycle events. Example: session created, run started, model routed.
  WARNING  Recoverable issues. Example: provider health degraded, rate limit near.
  ERROR    Failures that affect the user. Example: callable failed, policy denied.

25.2 Loggers (all via structlog)
  citnega.runtime        Runtime lifecycle events
  citnega.adapter.<fw>   Framework adapter events
  citnega.model_gateway  Model routing, rate limiting
  citnega.storage        Repository operations
  citnega.kb             KB ingestion and search
  citnega.security       Key store, scrubber actions
  citnega.tui            TUI worker events
  citnega.cli            CLI command execution

25.3 JSONL App Log Format (logs/app/<date>.jsonl)
```json
{
  "schema_version": 1,
  "timestamp": "2026-04-08T12:00:00.000Z",
  "level": "info",
  "logger": "citnega.runtime",
  "event": "session_created",
  "session_id": "...",
  "extra_field": "..."
}
```

25.4 JSONL Event Log Format (logs/events/<run_id>.jsonl)
One file per run. Each line is a CanonicalEvent serialized to JSON. The
schema_version field lets replay tools handle forward-compatible changes.

25.5 Log Retention
  App logs:    rotated daily, compressed, deleted after logging.retention_days
  Event logs:  kept indefinitely by default; deleted with session on request
               (or cleaned by a future maintenance command)
  KB exports:  kept until user deletes

25.6 Trace Correlation
Every log record within a run includes run_id and session_id. Every callable
invocation includes invocation_id. The TUI's /trace command queries
callable_invocations table filtered by run_id and displays the tree.

25.7 LogScrubber Strategy
Scrubber uses both allowlist and denylist:
  - Deny patterns (field names): api_key, token, secret, password, authorization,
    credential, bearer, auth
  - Values matching common secret patterns (high-entropy strings > 20 chars in
    denied field names) are redacted to "***REDACTED***"
  - Scrubber runs as a structlog processor on every log record before writing.
  - Same scrubber applied to event log writer (EventEmitter uses it too).

26. SECURITY AND SECRETS
================================================================================

26.1 API Key Storage
  Primary:   keyring (macOS Keychain, Windows Credential Manager, Linux Secret Service)
  Fallback:  environment variables (CITNEGA_<PROVIDER>_API_KEY)
  Never:     plaintext config files, JSONL logs, tracebacks

26.2 Setting Keys
```
citnega config set-key openai CITNEGA_OPENAI_API_KEY
# Prompts user for value, stores in keyring.
```

26.3 Retrieving Keys
Only ModelGateway and IModelProvider implementations call IKeyStore.get_key().
Keys are held in memory only for the duration of a request. Not logged.

26.4 File Permissions (Unix-like)
  <app_home>/               0700
  <app_home>/config/        0700
  <app_home>/db/            0700
  <app_home>/logs/          0700
  All files created within: 0600
Windows: default permissions are acceptable; keyring covers secret storage.

26.5 Session Isolation
  - Per-session IFrameworkRunner instance; no shared state.
  - Per-session asyncio.Queue; cross-session leakage impossible at queue level.
  - SQLite foreign keys ON prevents cross-session orphaning.
  - File paths in tool policies enforce per-session working directories if
    configured (allowed_paths can include ${SESSION_ID} substitution).

26.6 Tool Policy Defaults (restrictive)
  timeout_seconds       30
  requires_approval     true for write_file, delete_file, shell-like tools
  allowed_paths         empty by default (no file access)
  network_allowed       false by default
  max_output_bytes      256 KB
These are explicitly overridden in tool_registry.toml per tool.

26.7 Threat Model
Citnega assumes a trusted local user. It protects against:
  - Accidental data exposure (key leak to logs, policy bypass, cross-session leak)
  - Runaway tool execution (depth limits, timeouts, output caps)
  - Approval bypass (enforcement is centralized, not per-tool)
It does not protect against:
  - Malicious local users with shell access
  - Malicious third-party tools installed by the user
  - Supply-chain compromise of installed packages

27.2 Event Consumption Worker
```python
@work(exclusive=True, group="run_events")
async def consume_events(self, run_id: str) -> None:
    queue = self.app_service.runtime.get_event_queue(run_id)
    while True:
        event = await queue.get()
        self.call_from_thread(self._dispatch_event, event)
        if isinstance(event, RunCompleteEvent):
            break
```

27.3 Inline Block Dispatch
Events map to widget updates:
  TokenEvent              → TokenStreamBlock.append(token)
  CallableStartEvent      → ToolBlock/SpecialistBlock.create(mounted)
  CallableEndEvent        → ToolBlock/SpecialistBlock.finalize(result)
  ApprovalRequestEvent    → ApprovalBlock.create + focus
  ErrorEvent              → ErrorBlock.create
  RunStateEvent           → StatusBar.update(state)
  RunCompleteEvent        → worker exits

27.4 Headless Approval Mode (for CLI)
When running via CLI (Typer), there is no Textual app. Approvals are handled as:
  - Interactive mode (tty attached): print prompt to stdout, read from stdin.
  - Non-interactive mode: --auto-approve flag auto-approves, else auto-deny
    after a short warning log.
The CLI command `citnega run` prints ApprovalRequestEvents inline and pauses
for user input.

29. TESTING STRATEGY
================================================================================

29.1 Test Categories
  unit/           One test module per class; interfaces mocked
  integration/    Full stack with real SQLite, real filesystem, real adapters
  replay/         Deterministic replay tests from stored JSONL event logs
  adapters/       Shared LSP test suite — runs against ALL three adapters
  tui/            Textual app.run_test() based TUI tests
  property/       Hypothesis-based property tests for type invariants

29.2 Fixture Strategy
Shared fixtures in tests/fixtures/:
  db_factory()           Temp SQLite DB in tmpdir with WAL + migrations applied
  path_resolver()        Temp app home directory
  key_store()            In-memory IKeyStore for tests (no real keyring)
  mock_model_provider()  Deterministic IModelProvider that returns fixed responses
  mock_framework_runner() Deterministic IFrameworkRunner for unit tests of runtime
  registered_tools()     CallableRegistry pre-populated with builtin tools
  event_queue()          Fresh asyncio.Queue for event flow tests

29.3 Adapter LSP Suite
tests/adapters/shared_suite.py defines pytest tests that accept a fixture
`adapter: IFrameworkAdapter`. Each adapter package provides a conftest.py that
instantiates its adapter and provides it via the fixture. Tests run once per
adapter.

Example test cases:
  test_create_runner_returns_runner
  test_run_turn_emits_events_in_order
  test_pause_resume_cycles_correctly
  test_cancel_stops_execution
  test_checkpoint_roundtrip
  test_invokes_simple_tool
  test_invokes_specialist_with_nested_tools
  test_translates_unknown_event_to_generic
  test_policy_error_propagates_as_error_event
  test_depth_limit_enforced

Skipping an adapter's suite is only permitted if the framework package is not
installed (pytest skip marker), never for implementation reasons.

================================================================================
32. IMPLEMENTATION PHASES
================================================================================

Phase 1: Foundation and Contracts
  All protocol models, interfaces, errors, and infrastructure primitives.
  Everything covered by unit tests with mocks.

Phase 2: Runtime Core
  CoreRuntime, ContextAssembler, PolicyEnforcer, EventEmitter, CallableRegistry,
  AdapterRegistry, ModelRegistry. Tested with mock adapter and mock providers.

Phase 3: Framework Adapters (all three in parallel — equal treatment)
  ADK, LangGraph, CrewAI adapters. All three must pass the shared LSP test suite.
  Dependencies marked as optional in pyproject.toml.

Phase 4: Model Gateway
  IModelProvider impls (Ollama, OpenAI-compat, vLLM). Routing, rate limiting,
  token counting, health checks.

Phase 5: Agents and Tools
  All v1 tools, specialists, and core agents. Prompt templates. HybridRoutingPolicy.

Phase 6: Headless CLI
  Typer command tree. End-to-end validation of service layer without TUI.

Phase 7: TUI
  Textual app shell, inline widgets, event consumer worker, slash commands.

Phase 8: Knowledge Base (v1)
  KBStore, ingestion, FTS5 indexing, retrieval, export. KBRetrievalHandler.

Phase 9: Hardening
  Graceful shutdown, cross-platform path tests, checkpoint/recovery tests,
  replay tests, rate limiter tests, full LSP verification, packaging.

Phase 10: Future (post v1)
  KB v2 (entities, topics, embeddings), DuckDB analytics, FastAPI wrapper,
  user-defined plugins.

34.4 Distribution

================================================================================
35. FINAL PRINCIPLES AND EXECUTIVE SUMMARY
================================================================================

Framework Agnosticism
  ADK, LangGraph, and CrewAI are equal peers. Framework is a config value.
  Only the adapter layer imports framework packages. The shared LSP test suite
  validates that all three adapters are drop-in interchangeable.

Uniform Callable Structure
  Tools, specialists, and core agents share IInvocable. They differ only in
  which sub-interfaces they implement (ISP). Pre/post logic (tracing, events,
  policy) is centralized in BaseCallable via Template Method (DRY, SRP).

SOLID
  Every class has one reason to change (SRP). New behavior goes in new files
  (OCP). All implementations are substitutable (LSP). Interfaces are narrow
  and segregated (ISP). Dependencies flow toward abstractions (DIP); concrete
  wiring only in bootstrap.py.

DRY
  One canonical location for: pre/post callable logic (BaseCallable),
  context assembly (ContextAssembler), policy enforcement (PolicyEnforcer),
  path construction (PathResolver), log scrubbing (LogScrubber), WAL init
  (DatabaseFactory), registry operations (BaseRegistry), event base fields
  (BaseEvent).

Infrastructure
  SQLite WAL mode from first connection. Alembic migrations from day one.
  platformdirs via PathResolver. keyring for secrets. structlog JSONL with
  schema_version. uv workspaces. Graceful shutdown with checkpoint preservation.

Testing
  Unit coverage ≥90% on runtime, ≥100% on protocol. All adapters pass the
  same LSP integration suite. Hypothesis for type invariants. Replay tests
  from JSONL event logs. Cross-platform CI matrix (Linux, macOS, Windows).

Build Order
  Contracts first (Phase 1). Runtime core with mocks (Phase 2). All three
  adapters in parallel (Phase 3). Model gateway (Phase 4). Agents and tools
  (Phase 5). Headless CLI (Phase 6). TUI (Phase 7). KB v1 (Phase 8).
  Hardening (Phase 9).

Executive Summary
  Citnega is a local-first, Python-first, general-purpose agentic platform
  with a conversational terminal UI modeled after Gemini CLI and Claude Code.
  It runs on Linux, macOS, and Windows. The defining property is framework
  equality: ADK, LangGraph, and CrewAI are interchangeable config-selected
  runtimes. Agents are divided into core agents (orchestration) and specialist
  agents (domain tasks), both sharing the same uniform callable structure as
  tools. The architecture is designed under SOLID and DRY throughout, with
  every interface signature, Pydantic model, SQL schema, TOML config, error
  type, state machine, and phase checklist defined in this document.

  A competent Python engineer should be able to start coding from this spec
  without needing to infer any structural decisions.

End of specification.
